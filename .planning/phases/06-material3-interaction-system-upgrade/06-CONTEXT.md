# Phase 6: Material 3 交互系统化升级 - Context

**Gathered:** 2026-03-31
**Status:** Ready for planning
**Source:** Interactive discussion + codebase evidence review

<domain>
## Phase Boundary

在已完成 Phase 04/05 视觉迁移与细节修复的基础上，本阶段聚焦“交互系统化升级”：以 Material Design 3 为主线，统一状态反馈、层级与 token 语言、可访问性与触达反馈，并在不牺牲工具效率的前提下适度引入 md3E 表达。该阶段不引入新产品能力，不进行信息架构重构，不做脱离 MD3 的全面风格重做。

**讨论优先级更新（用户最新要求）**：本阶段后续讨论以“交互逻辑优化”为核心，文案细节不再作为主要讨论对象；在不影响理解的前提下沿用已锁定文案。

</domain>

<decisions>
## Implementation Decisions

### 状态反馈策略（优先级最高）
- 首页首次进入与切月采用“摘要+日历全覆盖”骨架屏方案，而非静默 placeholder。
- 骨架屏动画节奏采用“慢一点更稳”的速度策略，避免高频使用中的视觉疲劳。
- 空月份采用“引导型空态”：明确这是空数据状态，并将“去录入当天”作为主 CTA（去设置时薪作为次级引导）。
- 空月份主标题文案固定为“暂无数据”，主按钮文案固定为“去记今天”。
- 日历编辑采用“关闭即自动保存”交互（点击 sheet 外部关闭时触发）；保存成功后使用顶部小条静态提示“修改已保存”（低打扰可持续可见）。
- 自动保存边界：未改内容时不保存；改为 0 时按“清空当天记录”处理。
- 自动保存触发时机固定为“点击外部关闭 sheet 时触发”，不采用“每次改动自动写入”或“停手延迟写入”。
- 手势分工：下滑关闭走“取消路径”（不保存），并提供轻提示帮助用户建立操作习惯。
- 保存链路需优化为快速完成路径：默认不允许出现“后台仍在保存但同一天已可再次编辑”的竞态体验。
- 自动保存失败：顶部红色短提示，文案固定为“保存失败，点这里重试”，并提供可重试入口。
- 重试策略：用户点击重试后，系统自动补一次重试（最多再试 1 次）。
- 若达到重试上限仍失败：保留本地草稿并标记为“待处理”，后续采用“自动重提 + 手动提交并存”策略，避免数据丢失。
- 若保存超过阈值（如 6 秒）且队列存在后续日期：提供“继续等待 / 放弃本次并转下一条”二选一分流。
- 撤销策略：自动保存完成后提供 2 秒“撤销”入口，允许用户快速反悔。
- 自动保存成功文案：使用“修改已保存”（显示在顶部小条）。
- 用户点击撤销后的反馈文案：使用“已取消保存”。
- 用户点击撤销后保持在当前页，不自动重开编辑 sheet。
- 若用户切月后仍在撤销窗口内点击撤销：在当前月完成回滚并静默刷新，不强制跳回原月。
- 自动保存失败红条不自动消失，直到用户处理（重试或主动关闭）。
- 同一天快速连点仅首点生效，其余点击在短时间窗口内忽略，避免重复打开与竞态。
- 若当前日期正在保存、用户又点了另一日期：新操作进入队列，待当前保存完成后自动打开目标日期编辑。
- 队列策略采用“最后意图优先”：保存期间若连续点多个日期，仅保留最后一次目标，保存完成后直达该日期。
- 切换月份时若存在未提交改动，流程固定为“先自动保存，再切月”，防止跨月操作导致修改丢失。
- 若“先保存再切月”中的保存失败：保持在当前月，不切走，并提供就地重试入口。
- 用户触发“外部关闭=自动保存”后若立刻离页（返回/切后台），需短暂阻塞至关键落盘完成，确保不丢改动。
- 若触发关闭保存后应用被系统杀掉或异常退出，重启时优先恢复待提交项并引导用户确认继续提交。
- 清空当天记录不做二次确认，保持高频操作直达。
- 备份/恢复/导出/检查更新等长任务采用“底部进度弹层”作为统一显式反馈形态；进行中允许最小化后台运行，并在完成后给出结果摘要。
- 长任务进行中允许进入编辑，但会与任务冲突的操作需临时禁用，避免互相覆盖状态。
- 被禁用的冲突操作需给出即时原因，并提供一键跳转到进度入口，减少“为什么点不了”的困惑。

### 设计系统统一策略（MD3-first）
- 整体密度目标：紧凑但透气（compact but breathable）。
- 形状语言：柔和统一，建立固定 shape token，而非继续扩散局部半径值。
- 层级语言：明确三层级（页面 / 容器 / 关键卡片），用 tonal + elevation 组合表达，而非依赖局部补丁。
- 按钮强调：以 tonal 为主，filled/error 仅用于真正关键或危险动作。
- spacing 基线采用“4dp 网格 + 关键 8 倍数”策略，兼顾精细调整与全局一致性。

### frontend-design 气质方向
- 视觉气质锁定为“精致工具感”（refined utility feel），强调高频使用下的稳定、清晰、可预期。
- 排版层级中“数字优先”：金额现状可维持，时长信息再上调一级可读性（小幅强化，不做极端对比）。
- 动效采用“适度弹性”：用于状态切换与反馈确认，不追求炫技。

### 高优先交互触达细节
- 日历 DayCard 点击反馈采用“tonal 按压 + 微缩放”组合；ripple 保持为语义底座，但以按压态提升“已点击”确认感。
- DayCard TalkBack 语义采用“日期 + 类型 + 时长”三段式；类型以高密度单字表达（工 / 休 / 节）。
- 首页常用操作按钮采用“下半区固定位置”策略，减少拇指移动距离。
- 高频按钮目标尺寸采用 52dp，兼顾易点与布局密度。
- 低频按钮不与主操作并排，收进二级区，主界面优先保障高频流程。
- 设置入口保留在右上角，但点击命中区扩大到 52dp，优先修复“看得到但不容易点到”的问题。
- DayCard 采用“整卡可点”策略，避免用户必须命中小热点。
- 编辑面板关键动作（关闭/撤销）位置固定在同一角落，形成稳定肌肉记忆。

### md3E 使用边界
- md3E 仅用于增强交互可感知性与层级清晰度（如轻弹性、强调层级、可扫描性），不改变产品为效率工具的核心定位。
- 禁止高噪声表达（过强动态、过度装饰、破坏信息密度的视觉戏剧化改造）。

### 长任务反馈文案与入口
- 长任务完成摘要采用“短句 + 结果数值”风格（例如：恢复完成，导入 128 条记录）。
- 恢复完成后显示“导入 X 条”，并直接跳回当月视图。
- 进度弹层最小化后，使用“顶部小条状态提示”作为统一回入口，保证可回看且不遮挡主内容。
- 多任务并发时，顶部小条默认展示“最新任务 + 其余任务总数”。
- 顶部小条状态提示采用“与页面同色 + 细边框”样式，降低打扰感同时保持可见性。
- 用户点击顶部小条后，先打开“任务列表面板”，再进入具体任务进度。
- 长任务进行中切后台再回前台时，自动恢复到当前进度状态，保证流程连续。
- 网络较慢且进度长时间停顿时，显示“网络较慢，仍在处理中”安抚提示。

### Claude's Discretion
- 骨架屏具体形态（摘要卡/日历的占位结构）与过渡节奏。
- 统一 token 的命名与落地形式（如 `Spacing`, `Shape`, `Elevation`, `ActionEmphasis`）。
- 轻动效参数（duration/spring stiffness/damping）与触觉反馈联动细节。
- 长任务反馈采用弹层还是嵌入式进度的最终组件形态（在不违背“显式进度”前提下）。

</decisions>

### Established Patterns
- 当前已是 Compose + Material3 单 Activity 架构，适合分批替换，不需要引入架构级重构。

- `ViewModel + StateFlow` 管线已成熟，可在不改变业务语义的前提下引入明确 UI 状态机。

### Integration Points
- 根链路：`MainActivity.kt` -> `Screens.kt` -> `OvertimeAppShell.kt` -> `OvertimeNavigation.kt`。
- 主页状态入口：`OvertimeViewModel.uiState` 与 `HomeScreen` 的状态映射。
- 长任务反馈入口：`OvertimeAppEffects.kt` 中 snackbar/launcher/confirm dialog 链路。

<specifics>
## Specific Ideas

- 用户明确偏好：
  - “状态反馈”优先：骨架屏、引导型空态、保存成功可感知、长任务显式进度。
  - “设计系统统一”优先：紧凑但透气、柔和统一、清晰三层级、按钮以 tonal 为主。
  - “frontend-design + md3E 方向”：精致工具感、适度弹性、数字更突出。
- 阶段策略：先系统化，再做轻量 expressive；避免再次进入大范围换皮路径。

</specifics>

<evidence_based_guardrails>
## 有依据的交互护栏（非个人风格）

以下原则用于约束本阶段实现，避免“凭感觉设计”：

- **状态要看得见**：用户操作后要尽快看到反馈（来自经典可用性原则“系统状态可见”）。
  - 落地：顶部状态小条、保存结果文案、网络慢安抚文案保留。

- **错误要可恢复**：错误提示必须告诉用户下一步怎么做（重试/修正），且不要丢用户输入。
  - 落地：`保存失败，点这里重试` + 一次自动补重试 + 持续可见直到处理。

- **加载要让人安心**：长加载优先骨架，不让用户面对“空白+不确定”。
  - 落地：首页“摘要+日历全覆盖”骨架；网络慢时明确提示“仍在处理中”。

- **高频操作优先省步数**：工具型场景要减少确认弹窗和重复点击。
  - 落地：关闭即自动保存；清空当天记录不做二次确认；保留 2 秒撤销入口作为安全网。

- **动效要服务理解，不是装饰**：动效只用来确认状态变化，不做花哨演出。
  - 落地：轻动效、慢一点更稳、顶部小条低打扰样式。
