# 04-06 Plan - Post-device light-theme hierarchy follow-up

## Objective

Merge the former `4.1` real-device light-theme hierarchy fix back into Phase 04 so later readers can follow one continuous migration story.

## Historical Context

- After 04-05, real-device validation showed that light-theme card hierarchy still felt too flat on home and settings surfaces.
- The first response briefly tried subtle border reinforcement, but the stable direction was tonal layer separation rather than border-first patching.
- This plan absorbs those follow-up iterations into the main Phase 04 record.

## Scope

- `app/src/main/java/com/peter/overtimecalculator/ui/theme/ThemeDefaults.kt`
- `app/src/main/java/com/peter/overtimecalculator/ui/HomeSummarySections.kt`
- `app/src/main/java/com/peter/overtimecalculator/ui/settings/SettingsCommon.kt`
- `app/src/main/java/com/peter/overtimecalculator/ui/settings/SettingsMainSections.kt`
- `app/src/test/java/com/peter/overtimecalculator/ThemeDefaultsTest.kt`

## Tasks

1. Increase light-mode separation between `navigationContainer`, `sectionContainer`, `cardContainer`, and `cardElevatedContainer`.
2. Prefer semantic tonal hierarchy over black-border patching for summary/settings cards.
3. Keep dark-mode branch stable while strengthening light-mode readability.
4. Add luminance-delta guardrails to tests so future token tuning cannot flatten the hierarchy.
5. Validate on real devices in light mode after build/unit-test gates pass.

## Verification Gate

- `./gradlew.bat :app:assembleDebug --stacktrace`
- `./gradlew.bat :app:testDebugUnitTest --stacktrace`
- Real-device light-theme review for home summary/month switcher and settings cards

## Notes

- Historical source material remains in `.planning/phases/04.1-/` for audit trace, but this file is the preferred reading path for future Phase 04 maintenance.
- This follow-up remains visual-only. No business behavior, callback contracts, or navigation semantics should change.
