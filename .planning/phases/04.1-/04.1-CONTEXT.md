# 04.1 Context - 亮色卡片层次修复（URGENT）

## Trigger

真实机反馈：在亮色主题下，卡片边界和分层感明显不足；暗色主题表现正常。  
受影响范围主要是首页卡片（Summary、Month Switcher、Calendar cell）和设置页卡片容器的层次感。

## Goal

在不改业务行为的前提下，修复亮色主题的卡片可见性与层次感问题，确保用户在亮色模式下能快速识别分组关系和可交互区域。

## Proposed Improvement Plan

1. **Token tuning for light mode**
   - 调整 `ThemeDefaults` 在亮色模式下的层级差值（`navigationContainer` / `sectionContainer` / `cardContainer` / `cardElevatedContainer`）。
   - 优先提高容器之间的亮度差与边框可感知度，避免大面积同亮度区域。

2. **Card hierarchy by tone, not border patch**
   - 通过 `ThemeDefaults` 亮色层级分色（navigation/section/card/elevated）直接拉开卡片与背景差异。
   - 不依赖“黑色描边”作为主修复路径，避免 UI 风格回退到线框态。
   - 保证暗色模式不被过度提亮、不产生“灰雾感”。

3. **Text contrast stabilization**
   - 统一弱化文本色策略，避免亮色下副文本过浅导致“信息消失感”。
   - 校验金额、标题、副标题、状态文案在亮色背景上的对比度。

4. **No behavior changes**
   - 不调整交互语义、数据流与回调，仅改视觉 token 与容器呈现。

## Acceptance Targets

- 亮色首页：Summary 卡、月份切换卡、日历格子分层清晰，卡片边界可见。
- 亮色设置页：分组卡和条目层级清楚，说明文案可读。
- 暗色模式无明显回归。
- 构建与单测门禁通过；真机亮色截图复测通过。

## Risks

- 仅靠 alpha 微调可能不足，需要同步微调 surface tier 与 primaryContainer 混色比例。
- 若局部硬编码边框过多，后续维护成本上升，应优先通过语义 token 收敛。
